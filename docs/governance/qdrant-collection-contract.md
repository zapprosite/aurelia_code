**Purpose:** Definir schema de vetores, campos payload, e garantias operacionais para Qdrant

## Vector Generation

### Embedding Model: bge-m3

```
Model: BAAI/bge-m3
Dimensions: 384
Training Data: Multilingual (100+ languages)
Max Input Tokens: 8192
Format: HuggingFace transformers

Rationale:
- ✅ Local inference only (no API calls, offline-safe)
- ✅ Multilingual (Aurelia bot supports multiple languages)
- ✅ 384-dim is standard for semantic search (good latency/quality balance)
- ✅ Competitive with OpenAI text-embedding-3-small in many benchmarks
- ✅ MIT licensed (no proprietary restrictions)
- ✅ Fits in RTX 4090 VRAM budget (<2GB during inference)
```

### Encoding Process

```python
# Pseudocode: How vectors are generated
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('BAAI/bge-m3')
# Initialization: ~500ms, first load only

# For each markdown file in memory sync:
with open(filepath, 'r') as f:
    text = f.read()

    # Chunk if > 8192 tokens
    chunks = chunk_text(text, max_tokens=8192)

    # Generate vectors
    vectors = model.encode(chunks, normalize_embeddings=True)
    # Output: list[list[float]] (384-dimensional each)

    # Store in Qdrant with metadata
    upsert_to_qdrant(
        vectors=vectors,
        payloads=[{
            'chunk_index': i,
            'file_path': filepath,
            'memory_type': detect_type(filepath),
            'owner': extract_owner(filepath),
            'synced_at': datetime.now(),
            'text_content': chunk_text
        } for i, chunk_text in enumerate(chunks)]
    )
```

## Collection Configuration

### Qdrant Configuration (collection/config)

```yaml
vector_size: 384
distance: Cosine

storage_config:
  # Persistence strategy
  type: mmap # Memory-mapped for performance
  in_memory: false # Disk-backed for durability

  # WAL (Write-Ahead Logging) for durability
  wal_capacity_mb: 128
  wal_segments: 4

  # Snapshots for backup
  snapshots_path: /root/.qdrant/snapshots

quantization_config: null
  # Note: Do NOT use quantization (would degrade semantic quality)
  # Full precision 384-dim is requirement for Aurelia bot accuracy
```

### Performance Targets

| Metric | Target | Rationale |
|--------|--------|-----------|
| **Vector Write Latency** | <100ms | Batch operations (5min cron) |

| **Index Size** | <5GB | RTX 4090 VRAM allocation |
| **Recall@10** | >95% | Semantic search accuracy |
| **Point Count** | 100M+ possible | Memory scalability headroom |

## Operational Guarantees

### Durability
- ✅ Write-Ahead Logging (WAL) ensures no data loss on crash
- ✅ Snapshots every 6 hours to `/root/.qdrant/snapshots/`
- ✅ ZFS backing for container storage layer

### Availability
- ✅ Single node (no failover, acceptable for non-critical)
- ✅ Auto-restart via Docker policy (`restart: unless-stopped`)
- ✅ Health check: `curl localhost:6333/health` (required 2xx)

### Consistency
- ✅ Single writer (memory-sync-fiscal.sh cron)
- ✅ ACID-like for point operations (upsert is atomic)
- ✅ No concurrent writers (serialized by cron)

### Performance
- ✅ Index built on write (no batch re-indexing needed)
- ✅ Cosine similarity optimized for 384-dim vectors
- ✅ Query latency <50ms for typical workloads

## Sync Flow (Memory → Qdrant)

```
~/.claude/projects/-home-will-aurelia/memory/
    ↓
    [memory-sync-fiscal.sh --mode fast (every 5min)]
    ↓
    1. Scan *.md files
    2. Read YAML frontmatter (type, owner, tags)
    3. Split text into 8192-token chunks
    4. Encode with bge-m3 → 384-dim vectors
    5. Store in Postgres (ai_context.memory_entries)
    ↓
    [memory-sync-fiscal.sh --mode postgres-index (every 15min)]
    ↓
    6. Query Postgres for new/modified entries
    7. Upsert to Qdrant (repository_memory collection)
    8. Update synced_to_qdrant flag in Postgres
    ↓
    [Aurelia bot queries]
    ↓
    9. User question → encode with bge-m3
    10. Semantic search in Qdrant (cosine similarity)
    11. Retrieve top-K matches with payloads
    12. Inject into prompt context for LLM
    13. Response generated by qwen3.5:9b
```

## Constraints & Limits

### Hard Constraints
- ✅ Vector dimension MUST be exactly 384 (bge-m3 output)
- ✅ Distance metric MUST be Cosine (semantic meaning)
- ✅ No quantization (full precision required)
- ✅ Single collection `repository_memory` (no sharding)

### Soft Constraints
- ⚠️ Max payload per point: 100KB (typical: 8KB)
- ⚠️ Max points in memory: 100M (scalable to disk)
- ⚠️ Sync latency: Must complete within 5min window

### Breaking Changes
These changes would require ADR amendment:
- ❌ Change embedding model (would invalidate all vectors)
- ❌ Change vector dimension (incompatible with bge-m3)
- ❌ Add new indexed fields (requires schema migration)


### Alert Rules

```yaml
- alert: QdrantDown
  expr: aurelia_qdrant_available == 0
  for: 5m
  severity: critical
  action: Telegram alert, retry memory-sync

- alert: MemorySyncStale
  expr: (now - aurelia_memory_sync_duration_seconds) > 3600
  severity: warning
  action: Check memory-sync-fiscal.sh logs

- alert: CollectionSizeExceeded
  expr: qdrant_points_total{collection="repository_memory"} > 50000000
  severity: info
  action: Monitor, prepare for optimization
```
